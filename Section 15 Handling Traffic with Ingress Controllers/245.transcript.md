#### 2. Enabling Ingress on Minikube

For local environments using **Minikube**, the setup is simplified to a single command:

```bash
minikube addons enable ingress
```

* This enables the Ingress-NGINX controller inside your Minikube cluster.
* The output should confirm:

  ```
  ‚úÖ  ingress was successfully enabled
  ```

This command deploys:

* The **Ingress Controller** (running NGINX),
* A **default backend** for health checks,
* Associated **Service** objects to manage traffic internally.

---

#### 3. Comparison with Google Cloud (Preview)

* On **Google Cloud (GKE)**, enabling ingress automatically triggers the creation of:

  * A **Service of type LoadBalancer** inside the cluster.
  * An **external Google Cloud Load Balancer (GCLB)** resource.

This means that even though Ingress is the ‚Äúmodern‚Äù approach for routing,
**a LoadBalancer service still exists behind the scenes** to bridge external traffic into the cluster.

To verify this behavior:

* Navigate to the **GCE (Google Compute Engine)** section of the documentation in the same repo.
* Review the GCE Ingress configuration YAML ‚Äî it explicitly defines:

  ```yaml
  kind: Service
  spec:
    type: LoadBalancer
  ```

This confirms that GKE relies on a LoadBalancer to route traffic through the Ingress Controller.

---

#### 4. Summary

* **Local (Minikube)** ‚Üí Enable ingress with one command (`minikube addons enable ingress`).
* **Cloud (GKE)** ‚Üí Automatically provisions a LoadBalancer service and a Google Cloud Load Balancer.
* Even modern **Ingress-NGINX** setups internally depend on **LoadBalancer-type services** for external access.
